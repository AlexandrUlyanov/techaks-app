# ТЗ: Hero-зона главной страницы как управляемый промо-конструктор

Дата: 2026-06-04  
Проект: Techaks  
Статус: draft

## 1. Контекст

На главной странице Techaks уже есть hero-зона, а также отдельный альтернативный компонент `HeroInteractive`.

Однако текущий `HeroInteractive` не соответствует продуктовой задаче:

- он работает как декоративный fullscreen-экран;
- в нём нет динамических карточек товаров справа;
- нет полноценного управления содержимым hero из админки;
- нет ручного/автоматического режима подбора товаров;
- нет fallback-логики по товарам;
- композиция не соответствует схеме "слева торговое предложение, справа промо-карточки товаров".

То есть сейчас существует **визуальный альтернативный hero**, но не существует **hero как управляемого промо-конструктора витрины**.

При этом уже важно сохранить текущий hero как безопасную базовую версию. Пользователь должен иметь возможность в админке **всегда вернуться к текущему варианту hero**, который используется сейчас.

## 2. Цель

Сделать hero-зону главной страницы полноценным управляемым блоком витрины:

- с сильным торговым предложением;
- с динамическими карточками товаров из каталога;
- с редактируемыми преимуществами;
- с ручным и автоматическим режимом наполнения;
- с fallback-логикой;
- с управлением из админки;
- с возможностью в любой момент вернуть **текущую классическую версию hero**.

## 3. Ключевой принцип

Hero должен быть не просто красивым экраном, а **главным промо-конструктором главной страницы**.

Он должен решать реальные storefront-задачи:

- вести в каталог;
- вести в категории и подборки;
- показывать акционные/хитовые/рекомендованные товары;
- работать как управляемый коммерческий блок;
- не зависеть от внешних мессенджеров и внешних call-to-action.

Все действия hero должны вести **внутрь сайта**:

- в каталог;
- в категорию;
- в подборку;
- в акцию;
- в карточку товара;
- на страницу оплаты/доставки/самовывоза.

## 4. Что не делать

В рамках этой задачи нельзя:

- делать hero purely decorative без товарного слоя;
- превращать hero в рекламный баннер без связи с каталогом;
- зашивать преимущества и карточки в код;
- привязывать hero к WhatsApp, Telegram или другим внешним каналам;
- ломать текущую классическую hero-версию без возможности вернуться к ней.

## 5. Целевая структура hero

### 5.1. Desktop

Hero состоит из двух основных зон.

#### Слева:

- H1-заголовок;
- подзаголовок;
- основная CTA-кнопка;
- дополнительная CTA-кнопка;
- короткий список преимуществ.

Пример:

- H1: `Техника и аксессуары в Пензе — в наличии уже сегодня`
- Subtitle: `Смартфоны, наушники, зарядки, кабели, чехлы, power bank и гаджеты для дома, авто и работы. Выбирайте товары онлайн, проверяйте наличие и оформляйте заказ на сайте.`
- CTA 1: `Смотреть каталог`
- CTA 2: `Хиты продаж`

#### Справа:

- динамические карточки товаров из каталога;
- 3–5 карточек;
- одна карточка может быть крупнее, остальные компактнее;
- вся зона выглядит как аккуратная промо-раскладка, а не как обычная сетка каталога.

### 5.2. Mobile

Порядок блоков:

1. Заголовок
2. Подзаголовок
3. CTA-кнопки
4. Преимущества
5. Горизонтальный скролл карточек товаров

Требования:

- не делать слишком высокий первый экран;
- карточки прокручиваются горизонтально;
- кнопки крупные и удобные;
- текст сразу читается;
- без визуального перегруза.

## 6. Обязательная совместимость с текущей версией hero

### 6.1. Базовая версия

Текущая hero-зона должна быть зафиксирована как:

- `classic`
- или `legacy_classic`

Именно этот вариант должен считаться:

- безопасным;
- текущим боевым;
- доступным для мгновенного возврата из админки.

### 6.2. Новая hero-зона

Новая управляемая hero-зона должна жить как отдельный вариант:

- `promo_dynamic`

Позже можно расширить список:

- `classic`
- `promo_dynamic`
- `seasonal`
- `campaign`

Но сейчас достаточно сделать архитектуру с минимум двумя режимами:

1. `classic`
2. `promo_dynamic`

## 7. Управление из админки

Новый раздел:

`Главная страница → Hero-зона`

или внутри админки:

`Настройки/Главная → Hero-зона`

Если в проекте уже используется централизованная логика блоков главной, нужно встроиться в неё. Если нет — допустим отдельный раздел hero-настроек.

### 7.1. Основные настройки hero

Поля:

- `enabled` — включить/выключить hero
- `variant` — версия hero:
  - `classic`
  - `promo_dynamic`
- `title`
- `subtitle`
- `primaryButtonEnabled`
- `primaryButtonText`
- `primaryButtonUrl`
- `secondaryButtonEnabled`
- `secondaryButtonText`
- `secondaryButtonUrl`
- `openPrimaryInNewTab`
- `openSecondaryInNewTab`
- `sortOrder` — если главная управляется блоками

### 7.2. Управление преимуществами

Hero-преимущества должны редактироваться из админки как массив:

Каждый элемент:

- `text`
- `icon` (если используется внутренняя система иконок)
- `isActive`
- `sortOrder`

Значения по умолчанию:

- `В наличии в Пензе`
- `Гарантия на товары`
- `Актуальные цены`
- `Доставка и самовывоз`

### 7.3. Управление товарами hero

Нужно поддержать **два режима наполнения**:

1. `manual`
2. `automatic`

## 8. Ручной режим hero

В ручном режиме администратор сам выбирает карточки.

Для каждой hero-card:

- `productId`
- `badgeType`
- `customBadgeText`
- `customSubtitle`
- `showPrice`
- `showOldPrice`
- `showActionText`
- `actionText`
- `sortOrder`
- `isActive`

Вся карточка должна вести на страницу товара.

## 9. Автоматический режим hero

В автоматическом режиме hero получает товары по правилам.

Поддерживаемые источники:

- `hit`
- `recommended`
- `new`
- `promo`
- `popular`
- `discount`
- `in_stock`
- `category`
- `brand`
- `mixed`

Поля авто-режима:

- `source`
- `itemsLimit`
- `categoryId` / `categorySlug` (если источник category)
- `manufacturerId` / `manufacturerSlug` (если источник brand)
- `onlyInStock`
- `onlyWithPhoto`
- `onlyWithPrice`
- `excludeOutOfStock`
- `sortBy`

Допустимые `sortBy`:

- `popularity`
- `newest`
- `discount`
- `price_asc`
- `price_desc`
- `hero_sort_order`

## 10. Поля товара для hero

В модели товара и/или связанной storefront-проекции нужны поля hero-уровня:

- `isHit`
- `isRecommended`
- `isNew`
- `isPromo`
- `isPopular`
- `showInHero`
- `heroBadge`
- `heroSortOrder`
- `heroCustomTitle`
- `heroCustomSubtitle`
- `heroImage`

Если часть этих полей уже существует, переиспользовать.

Если нет — добавлять минимально и осмысленно.

## 11. Бейджи hero

Стандартные типы:

- `Хит продаж`
- `Рекомендуем`
- `Новинка`
- `Акция`
- `Выбор ТЕХАКС`
- `В наличии`
- `Выгодная цена`
- `Товар дня`
- `Популярное`
- `Скидка`

Также нужен кастомный текст бейджа.

## 12. Fallback-логика

Если товаров для hero нет или выборка некорректна:

1. брать товары с `showInHero = true`
2. затем `isHit = true`
3. затем `isRecommended = true`
4. затем популярные товары
5. затем последние активные товары с фото и ценой
6. если товаров всё равно нет — скрыть правую товарную часть, но оставить левую текстовую

Hero не должен показывать:

- пустые карточки;
- неактивные товары;
- товары без фото при включённом фильтре по фото;
- товары без цены при включённом фильтре по цене;
- товары без остатка, если включён фильтр in-stock.

## 13. Визуальный дизайн

### 13.1. Общий стиль

Hero должен выглядеть:

- современно;
- аккуратно;
- дорого;
- технологично;
- без тяжёлых теней;
- без шумных декоративных эффектов;
- без ощущения “игрового лендинга”.

### 13.2. Что важно сохранить

- много воздуха;
- сильный H1;
- понятные CTA;
- чистую сетку;
- акцент на товарах справа;
- фирменный стиль Techaks.

### 13.3. Что нужно убрать из текущего `HeroInteractive`

В новой реализации не должны быть центральной идеей:

- canvas как главный контент;
- декоративный курсор;
- сканирующие линии;
- абстрактные частицы как основная ценность hero;
- полноэкранный имиджевый режим вместо товарного промо.

Лёгкие анимации и фоновая динамика допустимы, но hero должен в первую очередь продавать и направлять пользователя по каталогу.

## 14. Карточки товаров в hero

Каждая карточка должна содержать:

- бейдж;
- фото;
- название;
- цену;
- старую цену, если есть;
- короткую подпись, если указана;
- ссылку на товар.

Требования:

- карточка целиком кликабельна;
- адаптивна;
- не должна ломать ритм первого экрана;
- на desktop может быть один акцентный товар и несколько компактных;
- на mobile — горизонтальный скролл.

## 15. Производительность

Hero — часть первого экрана, поэтому:

- не использовать тяжёлые библиотеки ради эффектов;
- не тащить в hero весь каталог;
- сервер/клиент должны получать только нужный набор hero-данных;
- использовать оптимизированные изображения;
- у карточек hero должен быть предсказуемый layout;
- lazy-loading допустим только для второстепенных карточек, не для критичного контента первого экрана.

## 16. Загрузка и пустые состояния

Нужны:

- skeleton для карточек;
- стабильная высота блока;
- плавное появление;
- отсутствие layout shift.

Если hero отключён — блок не показывается.

Если заголовок/настройки заполнены некорректно:

- storefront не должен показывать сломанный блок;
- в админке должно быть предупреждение о некорректной настройке.

## 17. Архитектура реализации

### 17.1. Frontend storefront

Нужно разделить:

- `HeroClassic` — текущая версия
- `HeroPromoDynamic` — новая управляемая версия
- `HomeHero` — маршрутизатор варианта

### 17.2. Backend

Нужен отдельный контур hero-настроек:

- получение public hero data для storefront;
- admin settings CRUD для hero;
- admin settings CRUD для преимуществ;
- admin settings CRUD для ручных карточек;
- сервис автоматической выборки товаров.

### 17.3. Admin preview

Желательно предусмотреть preview:

- desktop preview
- mobile preview

или хотя бы явный summary:

- какой режим включён;
- какие карточки реально попадут в hero;
- какие карточки были отброшены из-за отсутствия цены/фото/остатка.

## 18. Стадии реализации

### H1. Foundation

- выделить текущий hero в отдельный `classic` вариант;
- оформить новый data contract hero;
- создать admin settings для hero;
- создать storefront selector между `classic` и `promo_dynamic`.

### H2. Admin content model

- добавить hero-настройки;
- добавить преимущества;
- добавить ручной выбор карточек;
- добавить режимы наполнения.

### H3. Storefront promo hero

- собрать новый `HeroPromoDynamic`;
- desktop layout;
- mobile layout;
- skeleton/fallback.

### H4. Product metadata + auto-selection

- hero-поля товара;
- auto-source rules;
- fallback logic;
- badge rules.

### H5. QA + rollout

- preview;
- validation warnings;
- perf check;
- rollback to classic;
- финальная выкладка.

## 19. Acceptance criteria

Работа считается выполненной, если:

- текущая hero-версия сохранена и к ней можно вернуться из админки;
- hero управляется из админки;
- можно менять тексты, CTA и преимущества;
- hero умеет ручной и автоматический режим;
- hero показывает реальные карточки товаров справа;
- mobile hero имеет горизонтальный скролл карточек;
- не показываются невалидные карточки;
- есть fallback;
- hero не ломает производительность первого экрана;
- новая hero-зона соответствует торговой задаче, а не просто декоративному fullscreen-экрану.

## 20. Результат

В итоге Techaks должен получить не альтернативную “анимированную заставку”, а **реальный управляемый промо-блок главной страницы**, который:

- усиливает первый экран;
- помогает продавать;
- ведёт пользователя в каталог и карточки товара;
- использует данные каталога;
- контролируется из админки;
- и при этом всегда позволяет безопасно откатиться к текущему классическому hero.
