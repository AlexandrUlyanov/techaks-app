# Rollout checklist: бонусная программа МойСклад

Этот чеклист нужен, чтобы аккуратно выкатывать и проверять бонусную программу, где первоисточник правил, участников и фактических бонусных движений находится в МойСклад.

## 1. Предусловия

- В `.env` и production-секретах задан рабочий `MOYSKLAD_TOKEN`.
- На production применена миграция `db/migrations/0032_loyalty_jobs_and_snapshots.sql`.
- В таблице `app_settings` доступны настройки:
  - `loyalty_enabled`
  - `loyalty_group_name`
  - `loyalty_participant_tag`
  - `loyalty_default_max_writeoff_percent`
- В МойСклад существует группа участников бонусной программы `техакс`.
- Правила списания и акции в POS/лояльности МойСклад настроены и считаются главным источником правды.

## 2. Настройка в админке

Открыть: `Админка -> Лояльность`

Проверить:

- бонусная программа включена;
- группа по умолчанию: `техакс`;
- тег участника: `техакс`;
- fallback-ограничение на списание задано как резервное значение, но не спорит с правилами МойСклад.

## 3. Проверка очереди и воркера

Проверить:

- страница `Админка -> Лояльность` открывается без ошибок;
- в блоке очереди нет постоянного роста `pending/error`;
- в `pm2 logs techaks` нет циклических ошибок:
  - `loyalty_sync_jobs doesn't exist`
  - `loyalty-sync queue cycle failed`
  - `loyalty-maintenance schedule cycle failed`

Если есть застрявшие задачи:

1. открыть раздел лояльности;
2. нажать ручной прогон обслуживания;
3. проверить, уменьшается ли очередь;
4. если нет — смотреть `lastError` у job и `loyaltyLastSyncError` у заказа.

## 4. Проверка пользователя

Для тестового пользователя:

- в БД есть `moyskladCounterpartyId`;
- у пользователя в админке/разделе лояльности статус не `error`;
- группа/тег участника в МойСклад выставлены как `техакс`;
- у пользователя подтянулись:
  - баланс,
  - доступно к списанию,
  - pending accrual.

## 5. Проверка checkout

На checkout проверить три сценария.

### 5.1 Бонусная программа выключена

- блок бонусов не мешает оформлению;
- preview сообщает, что программа выключена;
- заказ оформляется без бонусной логики.

### 5.2 Пользователь не в группе `техакс`

- бонусный блок показывает понятное предупреждение;
- списание не применяется;
- заказ всё равно оформляется.

### 5.3 Пользователь участвует в программе

- видно баланс;
- видно максимум к списанию;
- сумма списания ограничивается правилами;
- итог заказа уменьшается;
- ожидаемое начисление считается от остатка после списания.

## 6. Проверка заказа после оформления

После оформления заказа с бонусами проверить:

- в заказе сохранены snapshot-поля:
  - `loyaltyBalanceBefore`
  - `loyaltyBonusRequested`
  - `loyaltyBonusSpent`
  - `loyaltyBonusExpectedAccrued`
  - `loyaltyPreviewPayloadJson`
  - `loyaltyRulesSnapshotJson`
- если было списание или ожидаемое начисление, у заказа статус синхронизации становится `pending` или `synced`.

## 7. Проверка админки заказа

Открыть `Админка -> Заказы -> конкретный заказ`.

Проверить блок `Бонусная программа`:

- баланс до заказа;
- запрошено к списанию;
- списано по факту;
- ожидалось к начислению;
- начислено по факту;
- статус синхронизации;
- время последней синхронизации;
- последняя ошибка, если она есть;
- кнопка `Пересинхронизировать бонусы` работает и обновляет данные.

## 8. Проверка фактов из МойСклад

После оплаты/завершения заказа:

- loyalty worker подтягивает фактическое списание и начисление;
- `loyaltyRawResultJson` обновляется;
- `loyaltyActualSpent` и `loyaltyActualAccrued` соответствуют данным из МойСклад;
- `bonus_transactions` содержат движения:
  - `loyalty_debit`
  - `loyalty_accrual`
  - при возврате: `loyalty_refund` и/или `loyalty_accrual_rollback`.

## 9. Негативные сценарии

Нужно отдельно прогнать:

### 9.1 Отмена неоплаченного заказа

- списание не фиксируется;
- начисление не создаётся;
- sync status -> `cancelled`.

### 9.2 Возврат/рефанд

- списанные бонусы возвращаются;
- начисленные бонусы сторнируются;
- sync status -> `rolled_back`.

### 9.3 Ошибка связи с МойСклад

- job уходит в `error`;
- ошибка видна в журнале;
- повторный прогон возможен без ручного вмешательства в БД.

## 10. Smoke после deploy

- `npm run build` проходит;
- `/api/health` отвечает `200`;
- админка лояльности открывается;
- checkout открывается;
- создание тестового заказа с бонусами проходит;
- карточка заказа в админке показывает бонусный блок без frontend-ошибок.

## 11. Что мониторить первые сутки

- рост `failedJobs` в разделе лояльности;
- повторяющиеся `timeout / ETIMEDOUT / network` по МойСклад;
- расхождения между:
  - `loyaltyBonusExpectedAccrued`
  - `loyaltyActualAccrued`
  - правилами бонусной программы в МойСклад;
- кейсы, где пользователь не попал в группу `техакс`.
