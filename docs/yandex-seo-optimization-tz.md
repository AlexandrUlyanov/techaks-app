# Yandex SEO Optimization TZ

Дата: 2026-06-02  
Проект: Techaks  
Статус: draft

## 1. Контекст

Techaks уже имеет базовый SEO-контур:

- есть `robots.txt`;
- есть `sitemap.xml` и дочерние sitemap для категорий, товаров, страниц и изображений;
- есть верификация Яндекса;
- подключена Яндекс Метрика;
- на storefront используется клиентский SEO helper для `title`, `description`, `canonical`, `robots`, `og:*`, `twitter:*`;
- на карточке товара уже есть JSON-LD `Product`;
- для Яндекса уже публикуется YML-фид товаров.

Однако текущая архитектура витрины остаётся SPA-first. Для Яндекса это создаёт несколько рисков:

- мета-теги и canonical выставляются на клиенте после загрузки JavaScript;
- важные страницы каталога индексируются менее стабильно, чем серверно-рендеренный HTML;
- canonical-логика для категорий сейчас слишком грубая;
- отсутствует единая SEO-система для категорий, брендов, магазинов и коммерческих страниц;
- отсутствует формализованный контроль качества индексации, сниппетов и коммерческих факторов в Яндекс Вебмастере.

## 2. Цель

Сделать так, чтобы storefront Techaks индексировался в Яндексе устойчиво и предсказуемо, а трафик из Яндекса рос за счёт:

1. корректной технической индексации;
2. правильной canonical/noindex политики;
3. серверно доступного SEO-контента на ключевых страницах;
4. сильных коммерческих и региональных сигналов;
5. качественных товарных, категорийных и брендовых страниц;
6. корректного Yandex YML feed и Яндекс.Метрики;
7. системного контроля через Яндекс Вебмастер и внутренние SEO-аудиты.

## 3. Главные проблемы на текущий момент

### 3.1. SPA-first SEO

Сейчас критичные SEO-данные для storefront в основном живут в React и проставляются через клиентский `useSeo`.

Это означает:

- HTML, который получает робот до выполнения JS, беднее, чем нужен для SEO;
- индексирование и обновление сниппетов в Яндексе могут быть менее стабильными;
- часть страниц зависит от того, насколько хорошо Яндекс отрендерит JS.

### 3.2. Неправильная canonical-логика для каталога

Категорийные страницы каталога сейчас меняют `title` и `description`, но canonical остаётся общим `/catalog`.

Последствия:

- все категории могут канонизироваться в один URL;
- Яндекс теряет сигнал, что это отдельные посадочные страницы;
- могут появляться конфликты между индексируемой страницей и canonical.

### 3.3. Недостаточно проработан SEO-контур категорий и брендов

Не хватает:

- управляемых SEO-полей;
- шаблонов `title`/`description`;
- intro-текстов категорий;
- нормальной SEO-архитектуры брендовых страниц;
- единой логики хлебных крошек и микроразметки.

### 3.4. Недостаточно выделены коммерческие факторы

Для Яндекса это критичный слой для интернет-магазина:

- доставка;
- самовывоз;
- гарантия;
- возврат;
- контакты;
- реквизиты;
- региональность;
- реальные магазины и адреса.

Сейчас часть этого уже есть, но не собрана в чёткий SEO-коммерческий контур.

### 3.5. Нет formal SEO observability

Нет общего процесса и внутренних инструментов, которые позволяют быстро понять:

- какие страницы индексируются;
- какие excluded;
- где canonical конфликтует;
- где нет meta description;
- где нет intro-текста;
- какие категории/товары плохо представлены для Яндекса.

## 4. Целевая архитектура SEO под Яндекс

Storefront должен перейти к модели:

1. server-friendly SEO pages для ключевых типов страниц;
2. корректные canonical/noindex правила;
3. server-generated или prerendered HTML для главных SEO-посадочных;
4. единая система SEO-метаданных;
5. полноценная микроразметка;
6. сильные коммерческие и региональные страницы;
7. наблюдаемость и аудит.

## 5. Область работ

В задачу входят:

1. технический SEO-фундамент;
2. категории / бренды / товары / магазины / контентные страницы;
3. коммерческие и региональные сигналы;
4. микроразметка;
5. Яндекс Вебмастер и Метрика;
6. SEO-админка и внутренний аудит.

В задачу не входит:

- полная контент-стратегия на месяцы вперёд;
- закупка ссылок;
- внешний PR;
- performance-оптимизация сайта в целом, кроме SEO-критичных мест.

## 6. Этап 0. SEO baseline и аудит

### Цель

Зафиксировать текущее состояние, чтобы мы не “улучшали вслепую”.

### Что нужно собрать

1. Индексация
   - сколько страниц в индексе Яндекса;
   - сколько исключено;
   - какие типы страниц индексируются лучше/хуже.

2. Технический SEO-срез
   - robots;
   - sitemap;
   - canonical;
   - status codes;
   - наличие structured data;
   - наличие title/description/h1.

3. Коммерческий SEO-срез
   - наличие страниц доставки, оплаты, возврата, оферты, контактов;
   - привязка магазинов и адресов;
   - полнота карточек товара.

4. Контентный срез
   - категории без intro;
   - товары без описания;
   - товары без бренда/артикула/изображения;
   - категории без meta.

### Артефакты этапа

- таблица baseline;
- список текущих SEO-рисков;
- список страниц, которые уже готовы к индексации;
- список страниц, которые лучше закрыть или канонизировать.

## 7. Этап 1. Технический SEO-фундамент

### 7.1. Canonical и noindex policy

#### Цель

Сделать так, чтобы Яндекс чётко понимал, какие URL являются основными.

#### Что нужно сделать

1. Для главной:
   - canonical = `/`

2. Для категорий:
   - canonical = `/catalog?cat={slug}`

3. Для брендов:
   - canonical = `/catalog?view=brands&brand={slug}`

4. Для фильтрованных/служебных вариантов:
   - `sort`, `layout`, `limit`, лишние query-параметры — canonical на базовую категорию;
   - при необходимости `noindex, follow`.

5. Для служебных страниц:
   - `/checkout`
   - `/payment/result`
   - `/account`
   - `/login`
   - `/search`
   оставить `noindex`.

6. Проверить:
   - одинаковую canonical-политику на клиенте и в HTML-first сценариях;
   - отсутствие канонизации категорий в общий `/catalog`.

#### Acceptance criteria

- категории и бренды имеют собственный canonical;
- служебные страницы не индексируются;
- canonical не конфликтуют с реально индексируемыми посадочными.

### 7.2. Robots и sitemap policy

#### Цель

Упростить роботу Яндекса обход сайта и убрать мусорные URL.

#### Что нужно сделать

1. Пересмотреть `robots.txt`:
   - убрать индексирование служебных и параметрических страниц;
   - сохранить `Clean-param`;
   - проверить, не блокируем ли лишнее.

2. Проверить sitemap:
   - товары только публичные;
   - без нулевой цены;
   - без нулевого остатка, если это бизнес-правило для SEO;
   - скрытые товары не попадают;
   - категории без товарного смысла не попадают, если они пустые;
   - `lastmod` использовать из реального `updatedAt`, где возможно.

3. При росте числа товаров:
   - сделать sitemap чанками (`products-1`, `products-2`, ...).

#### Acceptance criteria

- sitemap содержит только реально полезные SEO-URL;
- robots не конфликтует с индексацией категорий/товаров;
- пустые и мусорные страницы не попадают в индекс через sitemap.

### 7.3. SSR / prerender / hybrid rendering

#### Цель

Снизить зависимость SEO от клиентского JavaScript.

#### Что нужно сделать

Выбрать и внедрить стратегию:

1. prerender для:
   - главной;
   - категорий;
   - карточек товара;
   - брендов;
   - магазинов;
   - статей;
   - юридических страниц.

или

2. полноценный SSR/SSG-гибрид, если это оправдано по срокам.

#### Требования

- важный SEO-контент должен быть доступен в HTML как можно раньше;
- title/description/canonical/JSON-LD не должны полностью зависеть от post-hydration;
- решение не должно ломать текущий storefront.

#### Acceptance criteria

- главные SEO-страницы имеют серверно доступный head и основной контент;
- Яндекс получает пригодный HTML без необходимости полагаться только на JS.

### 7.4. HTTP и index hygiene

#### Что проверить и исправить

- единый хост: `https://techaks.ru`;
- корректные redirect между `www`/без `www`, `http`/`https`;
- отсутствие soft 404;
- корректные `404/410` для снятых страниц;
- отсутствие дублей со slash/no-slash;
- проверка `rel=canonical` и фактических redirect.

## 8. Этап 2. SEO-система для типов страниц

### 8.1. Категории

#### Цель

Превратить категории в нормальные SEO-посадочные.

#### Что нужно сделать

Для каждой категории добавить или использовать:

- `seoTitle`
- `seoDescription`
- `seoIntro`
- при необходимости `seoH1`

Если ручные поля не заполнены, использовать шаблоны.

#### Шаблон на старт

- `Название категории — купить в Techaks в Пензе, цены и наличие`
- `Название категории в интернет-магазине Techaks: актуальные цены, наличие в магазинах, самовывоз и доставка.`

#### Контент

Intro-текст:

- короткий;
- полезный;
- без SEO-портянок;
- 400–900 символов;
- с естественным упоминанием назначения раздела.

#### Acceptance criteria

- каждая индексируемая категория имеет корректный title и description;
- у важных категорий есть intro;
- canonical и h1 не конфликтуют.

### 8.2. Брендовые страницы

#### Цель

Сделать из страниц брендов отдельные поисковые посадочные.

#### Что нужно сделать

- полноценный `title`, `description`, `h1`;
- описание бренда;
- список товаров бренда;
- нормальные breadcrumbs;
- canonical на брендовый URL;
- schema, если уместно.

#### Acceptance criteria

- брендовые страницы индексируются как отдельные посадочные;
- они не канонизируются в общий каталог.

### 8.3. Карточки товара

#### Цель

Сделать товарные страницы максимально сильными для Яндекса.

#### Что нужно усилить

- уникальный title;
- осмысленный meta description;
- бренд;
- артикул / код товара;
- цена;
- наличие;
- самовывоз / доставка;
- реальные характеристики;
- магазины с наличием;
- отзывы, если они есть.

#### Acceptance criteria

- товарная страница содержит максимум полезных для Яндекса коммерческих сигналов;
- title и description не дублируются массово;
- structured data для товара корректна.

### 8.4. Магазины и контакты

#### Цель

Усилить региональность и доверие.

#### Что нужно сделать

- отдельные страницы магазинов или сильная сводная страница магазинов;
- адрес;
- телефон;
- режим работы;
- схема проезда;
- самовывоз;
- магазин должен быть связан с товарным наличием.

#### Acceptance criteria

- страницы магазинов полезны и индексируемы;
- региональность Techaks для Пензы понятна и пользователю, и Яндексу.

### 8.5. Юридические и коммерческие страницы

Обязательные страницы:

- Контакты
- Доставка и оплата
- Возврат и обмен
- Оферта
- Политика конфиденциальности
- Гарантия

#### Требования

- все доступны из футера;
- не пустые;
- не скрыты за noindex без причины;
- тексты читаемые, а не служебная заглушка.

## 9. Этап 3. Микроразметка и сниппеты

### 9.1. Product schema

Проверить и довести:

- `name`
- `image`
- `description`
- `sku`
- `brand`
- `offers.price`
- `offers.availability`
- `offers.url`

`aggregateRating` и `review` использовать только если отзывы реальные.

### 9.2. BreadcrumbList

Добавить JSON-LD `BreadcrumbList` для:

- категории;
- товара;
- статьи;
- брендовой страницы.

### 9.3. Organization / Store

Добавить или усилить:

- `Organization`
- `Store`
- адреса;
- телефон;
- режим работы;
- URL сайта.

### 9.4. Article schema

Для блога:

- `BlogPosting` / `Article`
- headline
- datePublished
- dateModified
- image
- author

## 10. Этап 4. Коммерческие и региональные сигналы для Яндекса

### Цель

Усилить то, что Яндекс особенно ценит у интернет-магазинов.

### Что нужно сделать

1. Региональность
   - закрепить регион в Яндекс Вебмастере;
   - в UI и контенте чётко показывать Пензу и магазины.

2. Прозрачность магазина
   - реквизиты;
   - контакты;
   - гарантии;
   - возврат;
   - способы оплаты;
   - доставка;
   - самовывоз.

3. Связь с наличием
   - если товар доступен в магазинах, это должно быть явно видно на карточке и в SEO-сигналах страницы.

4. Консистентность между сайтом и YML
   - цена;
   - наличие;
   - бренд;
   - изображения.

## 11. Этап 5. Яндекс Вебмастер и Метрика

### 11.1. Яндекс Вебмастер

Нужно настроить и вести:

- основное зеркало;
- регион;
- sitemap;
- мониторинг исключённых страниц;
- анализ проблем индексации;
- сниппеты;
- безопасные уведомления об ошибках.

### 11.2. Яндекс Метрика

Проверить:

- корректность установки;
- ecommerce через `dataLayer`;
- цели:
  - просмотр товара;
  - добавление в корзину;
  - начало checkout;
  - успешная оплата;
  - резерв;
  - отправка формы / внутреннего сообщения.

### 11.3. KPI

Считать как минимум:

- индексируемые категории;
- индексируемые товары;
- органический трафик из Яндекса;
- CTR в Яндексе по категориям;
- CTR в Яндексе по товарам;
- доля страниц без meta;
- доля страниц без schema;
- доля категорий без intro.

## 12. Этап 6. SEO-админка и внутренние инструменты

### Цель

Сделать SEO управляемым, а не “ручным только через код”.

### Что нужно добавить

1. Поля SEO для категорий:
   - title
   - description
   - intro
   - override h1

2. При необходимости поля для брендов.

3. SEO health dashboard / audit view:
   - товары без description;
   - товары без image;
   - товары без brand;
   - категории без intro;
   - пустые meta;
   - дубли slug/title.

4. Возможность быстро видеть:
   - индексируемая ли страница;
   - canonical;
   - robots policy;
   - дата последнего обновления.

## 15. Нефункциональные требования

- не ломать текущую витрину;
- не ломать storefront UX;
- не ухудшить производительность без причины;
- не закрыть случайно нужные страницы от индексации;
- не допустить конфликтов canonical и noindex;
- сохранять совместимость с текущим YML feed;
- не писать хардкодные SEO-данные в компоненты, если их нужно будет управлять через админку.

## 16. Критерии приёмки

Работа считается выполненной, если:

1. категории, бренды и товары имеют корректную canonical-политику;
2. sitemap и robots отражают целевую индексационную стратегию;
3. важный SEO-контент доступен Яндексу в server-friendly виде;
4. категории и бренды стали самостоятельными SEO-посадочными;
5. карточки товаров содержат сильные коммерческие и структурные сигналы;
6. на сайте есть полноценная микроразметка для товара, хлебных крошек и организации;
7. региональность и магазины усилены;
8. Яндекс.Метрика и Яндекс Вебмастер используются системно;
9. в админке есть SEO-управление и базовый аудит качества;
10. после rollout есть baseline/after metrics, по которым видно улучшение.
