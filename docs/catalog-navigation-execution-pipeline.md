## 1. Общий принцип исполнения

Работы нужно выполнять не как набор несвязанных UI-правок, а как последовательную перестройку каталога:

1. Сначала фиксируем архитектурный контракт.
2. Затем приводим к нему корень каталога.
3. Потом переносим тот же принцип на родительские и промежуточные категории.
4. После этого стабилизируем навигационные механики: compact mode, sidebar, accordion, search.
5. Далее выносим preview-данные в отдельный легкий источник.
6. В конце доводим визуал, secondary shelves, SEO, analytics, QA и rollout.

## 2. Последовательность этапов

### Этап A. Архитектурный контракт

Задача:

- #159 `CAT-301`

Цель:

- закрепить логику `children => navigation`, `leaf => product listing`;
- убрать все спорные места, где один и тот же slug может вести себя неоднозначно.

Результат этапа:

- каталог имеет единый режимный контракт;
- дальнейшие задачи не спорят между собой.

Без этого этапа не начинать массовую правку интерфейсов.

---

### Этап B. Эталонный root catalog

Задача:

- #160 `CAT-302`

Цель:

- сделать `/catalog?cat=all` эталонной навигационной страницей;
- использовать её как reference implementation для следующих уровней.

Результат этапа:

- готов шаблон navigation root;
- можно переносить паттерн на остальные категории.

Зависит от:

- #159

---

### Этап C. Промежуточные категории и режимы layout

Задачи:

- #161 `CAT-303`
- #162 `CAT-304`

Цель:

- перевести parent/intermediate категории на navigation mode;
- внедрить правило compact layout для малых разделов.

Порядок:

1. Сначала #161 — общий перенос parent/intermediate категорий на navigation model.
2. Затем #162 — упрощение сценария для разделов с малым числом подкатегорий.

Результат этапа:

- крупные разделы не открываются как случайный PLP;
- маленькие разделы не выглядят как перегруженный “большой каталог”.

Зависит от:

- #159
- желательно после #160

## 5. Definition of Done для milestone

Milestone считается закрытым, если одновременно выполнено всё ниже:

1. Корень каталога — навигационный экран.
2. Родительские и промежуточные категории с children не открываются как PLP.
3. Leaf categories открываются как нормальные product listings.
4. Для малых разделов работает compact layout.
5. Для больших разделов работает sidebar mode.
6. Поиск по категориям работает стабильно.
7. Navigation pages используют lightweight preview-данные.
8. Category cards унифицированы визуально.
9. Secondary shelves не ломают основной сценарий выбора категории.
10. SEO / accessibility / analytics доведены.
11. Пройден QA по breakpoint matrix и dark theme.

---

## 6. Риски

Основные риски:

1. Двойная логика каталога в старом коде.
2. Случайные регрессии в leaf-category PLP.
3. Перегрузка navigation pages полной товарной выборкой.
4. Повторное появление дублей категорий после локальных правок.
5. Слабый mobile UX, если desktop-паттерн просто “ужать”.

Как снижать:

- не трогать все уровни сразу без #159;
- проверять root / parent / intermediate / leaf отдельно;
- держать QA-матрицу по категориям и breakpoint-ам;
- не смешивать навигационный этап и промо-этап в одном PR без причины.

## 7. Рекомендуемый формат реализации

Лучше вести milestone короткими PR-волнами:

1. Architecture PR
2. Root catalog PR
3. Parent/intermediate categories PR
4. Sidebar + mobile PR
5. Preview/search PR
6. Visual + shelves PR
7. SEO/a11y/analytics PR
8. QA/rollout PR

Такой формат упростит:

- код-ревью
- rollback
- проверку регрессий
- деплой по этапам
