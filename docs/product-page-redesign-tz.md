# Product Page Redesign TZ

Дата: 2026-05-18  
Проект: ТЕХАКС  
Область: витрина / карточка товара  
Статус: draft / ready for implementation

## 1. Контекст

Карточка товара в ТЕХАКС уже функциональна:

- есть галерея;
- отображается цена;
- работает добавление в корзину;
- есть наличие по магазинам;
- выводятся характеристики;
- есть похожие товары;
- внедрён review layer.

Но текущая страница в [src/pages/ProductPage.tsx](</E:/work/ru/tehax/s/app/src/pages/ProductPage.tsx>) всё ещё ощущается как линейный стек блоков, а не как собранная product-detail experience.

Основные проблемы текущего UX:

1. правая колонка не собрана в decision panel;
2. блок доверия и сигналов покупки не держит стабильную иерархию;
3. наличие по магазинам информативно, но визуально шумное;
4. описание и характеристики одинаково “плоские” по важности;
5. reviews-section уже полезен, но всё ещё живёт отдельно от shopping-flow;
6. визуальный язык ТЕХАКС сильный, но карточке не хватает точечных motion-accent деталей;
7. при отсутствии рейтинга/отзывов страница требует специальных layout-состояний, чтобы сетка не “проваливалась”.

## 2. Цель редизайна

Сделать карточку товара:

- более понятной для принятия решения;
- визуально чище;
- лучше сбалансированной по иерархии;
- сильнее по trust-signals;
- готовой к масштабированию на разные типы товаров;
- пригодной для аккуратных SVG / micro-motion анимаций без тяжёлого визуального шума.

Главная product-цель:

`Карточка должна быстрее отвечать на вопрос "подходит ли мне товар и можно ли его спокойно купить прямо сейчас?"`

## 3. Product principles

Карточка товара для ТЕХАКС должна строиться на таких принципах:

1. **Decision-first**
   - сначала то, что помогает решить “покупать / не покупать”;
   - потом уже полные данные.

2. **Trust before detail**
   - наличие, гарантия, доставка, отзывы и сигналы доверия должны быть видны раньше, чем длинный текст.

3. **Technical clarity**
   - карточка техники не должна быть похожа на маркетинговый лендинг;
   - меньше декоративности, больше ясной пользы.

4. **Stable layout**
   - карточка не должна разваливаться от пустых состояний:
   - нет отзывов;
   - нет старой цены;
   - один магазин;
   - нет описания;
   - мало характеристик.

5. **Motion as assistance, not decoration**
   - анимации только там, где они усиливают понимание действия или статуса.

## 4. Текущее состояние по секциям

### 4.1 Верх страницы

Сейчас:

- breadcrumbs;
- крупный H1;
- галерея слева;
- колонка покупки справа.

Проблемы:

- в правой колонке слишком много отдельных блоков, но нет единого decision-panel слоя;
- при отсутствии рейтинга верх правой колонки визуально “проседает”.

### 4.2 Price / CTA

Сейчас:

- цена;
- oldPrice при наличии;
- primary CTA.

Проблемы:

- цена не объединена с trust/availability summary;
- вторичные действия не образуют явный CTA cluster.

### 4.3 Наличие

Сейчас:

- отдельные карточки магазинов;
- отдельная warning-strip о низком остатке.

Проблемы:

- блок визуально громоздкий;
- summary и details не разделены;
- важный смысл “где и сколько доступно” не считывается мгновенно.

### 4.4 Specs / description

Сейчас:

- описание;
- длинный линейный список характеристик.

Проблемы:

- нет разделения на ключевые и вторичные характеристики;
- сканирование карточки занимает слишком много внимания.

### 4.5 Reviews

Сейчас:

- есть пустое состояние;
- есть composer;
- есть summary/funnel, если отзывы присутствуют.

Проблемы:

- reviews ещё недостаточно интегрированы в основной product flow;
- пустое и раскрытое состояние требуют более устойчивой архитектуры.

## 5. Целевой UX-концепт

Предлагается redesign в формате:

`Product Decision Page`

### Общая структура

1. breadcrumbs
2. hero title block
3. above-the-fold product area:
   - gallery rail
   - decision panel
4. key product explanation:
   - value points
   - compact availability
   - quick specs
5. full description/specs
6. reviews
7. related products

## 6. Целевая структура карточки

## 6.1 Above the fold

### Левая колонка

- основное изображение товара;
- бейджи товара;
- бренд;
- при наличии нескольких изображений — thumbnails rail;
- hover/press zoom-lite без тяжёлой галерейной механики.

### Правая колонка

Новый слой: **decision panel**

Содержимое:

1. trust strip
2. price card
3. CTA cluster
4. compact stock summary
5. quick value points
6. support action

## 6.2 Trust strip

Должен занимать фиксированную зону сверху правой колонки.

Если у товара есть отзывы:

- рейтинг;
- count отзывов;
- verified review hint.

Если отзывов нет:

- не показывать фальшивый rating;
- показывать нейтральный trust-strip:
  - `Пока без отзывов`
  - `Гарантия X`
  - `Новый товар` или другой релевантный сигнал.

Критично:

- layout по высоте не должен “прыгать”;
- зона остаётся, меняется только содержимое.

## 6.3 Price card

Price card должен стать главным decision-блоком.

Состав:

- актуальная цена;
- старая цена и экономия, если есть;
- короткий availability hint:
  - `В наличии`
  - `Самовывоз сегодня`
  - `Доставка от 1 дня`
- delivery / pickup micro hint;
- primary CTA.

### CTA cluster

1. primary:
   - `Добавить в корзину`
2. secondary:
   - `В избранное`
3. tertiary:
   - `Задать вопрос`

Кнопки должны быть собраны в один визуальный ряд/группу, а не быть рассыпаны по странице.

## 6.4 Quick value points

Под price card нужен короткий блок из 3–4 value points:

- гарантия;
- самовывоз / доставка;
- проверка перед выдачей;
- поддержка в Telegram.

Формат:

- icon + title + short hint

Это должно быть:

- коротко;
- полезно;
- спокойно;
- без маркетингового крика.

## 6.5 Availability summary

Текущий блок наличия нужно разделить на 2 уровня:

### Level 1: summary

Например:

- `В наличии в 2 магазинах`
- `От 1 до 2 шт.`
- `Есть ограниченный остаток`

### Level 2: details

Раскрывающийся список магазинов:

- магазин;
- адрес;
- количество;
- локальный статус.

Низкий остаток лучше показывать:

- не отдельной длинной полосой ниже,
- а как встроенный warning-chip в summary.

## 6.6 Key specs

Нужно отделить:

- ключевые характеристики;
- полный список характеристик.

### Key specs

Показывать 4–6 самых важных для категории характеристик над fold.

Примеры:

- мощность;
- совместимость;
- тип подключения;
- ёмкость;
- материал;
- гарантия.

### Full specs

Остальное скрывать в:

- accordion;
- или блоке `Показать все характеристики`.

## 6.7 Description

Описание не должно быть выше decision-layer.

Оно должно идти ниже:

- after trust;
- after price;
- after key specs.

Если описание длинное:

- сначала 4–6 строк preview;
- далее `Показать полностью`.

## 6.8 Reviews

### Если отзывов нет

По умолчанию:

- показывать только компактный CTA block;
- не показывать большой reviews dashboard;
- не показывать заголовок `ОТЗЫВЫ О ТОВАРЕ`.

После клика по CTA:

- раскрывается полноценная review section;
- выполняется скролл к composer.

### Если отзывы есть

Показывать:

- summary panel;
- breakdown;
- feed;
- ответ магазина;
- composer.

### Review section must support

- стабильное пустое состояние;
- авторизованный / неавторизованный сценарий;
- якоря `#reviews` и `#review-composer`.

## 7. SVG / motion proposal

## 7.1 Общий принцип

Анимации должны быть:

- короткими;
- функциональными;
- не мешающими чтению;
- не постоянными в idle-состоянии.

## 7.2 Разрешённые анимации

### 1. Cart icon

При клике на CTA:

- лёгкий micro-bounce;
- короткий shine / travel accent.

### 2. Rating star

При hover на рейтинге:

- лёгкий fill shimmer;
- не постоянное мигание.

### 3. Map pin / stock icon

При открытии availability details:

- короткий pulse;
- stroke emphasis.

### 4. Shield / guarantee icon

Для trust blocks:

- draw-in stroke animation при первом появлении в viewport.

### 5. MessageCircle / Telegram CTA

На hover:

- subtle bounce;
- не сильнее, чем primary CTA.

### 6. Availability chip

Для low stock:

- мягкий pulse-dot;
- не анимировать весь блок.

## 7.3 Что запрещено

Не делать:

- большие Lottie animations;
- фоновые бесконечные motion эффекты;
- прыгающие карточки;
- мерцающие цены;
- анимировать весь product hero.

## 7.4 Технология

Предпочтительно:

- lucide icons;
- inline SVG;
- CSS transitions;
- при необходимости лёгкий `framer-motion`.

Не использовать тяжёлые animation runtimes без реальной необходимости.

## 8. Mobile UX

Карточка товара должна отдельно проектироваться для mobile, а не просто ужиматься.

### Mobile priorities

1. title
2. gallery
3. price
4. CTA
5. trust signals
6. stock summary

### Mobile requirements

- CTA должен быть сразу виден без длинного скролла;
- наличие должно быть сначала summary, а не полный список;
- характеристики — только ключевые до раскрытия;
- review empty state не должен занимать пол-экрана бессмысленным пустым пространством.

## 9. Desktop UX

Desktop должен дать ощущение “богатой, но собранной” карточки.

Требования:

- hero зона должна быть читаема за 3–5 секунд;
- price / CTA / trust должны считываться как единый decision cluster;
- reviews не должны визуально спорить с блоком покупки;
- related products не должны красть внимание у основной карточки.

## 10. Visual design constraints

Сохраняем существующий визуальный язык ТЕХАКС:

- акцент `#05C3D4`;
- тёмный заголовочный контраст;
- светлые surface;
- резкие uppercase heading patterns там, где они уже являются частью бренда.

Но:

- меньше пустых “дыр”;
- меньше случайно одинаково важных блоков;
- лучше вертикальный ритм;
- лучше плотность информации.

## 11. Required implementation phases

## Phase 1 — layout foundation

Изменения:

- перестроить hero layout;
- собрать decision panel;
- стабилизировать trust strip;
- выровнять верхнюю зону без отзывов;
- сделать compact empty review state.

Выход:

- новая иерархия без сильной анимации.

## Phase 2 — stock/specs refinement

Изменения:

- summary + expandable availability;
- key specs;
- full specs accordion;
- улучшенная подача описания.

## Phase 3 — motion polish

Изменения:

- cart icon micro-animation;
- trust icon line-motion;
- stock pulse details;
- hover polish CTA / rating / support actions.

## Phase 4 — responsive refinement

Изменения:

- mobile folding;
- tablet balancing;
- text-fit checks;
- interaction polish.

## 12. Acceptance criteria

Редизайн считается успешным, если:

1. правая колонка воспринимается как единый decision panel;
2. при отсутствии рейтинга нет пустого визуального провала;
3. наличие по магазинам сначала читается как summary, а потом как detail;
4. ключевые характеристики видны раньше полного списка;
5. reviews без отзывов не шумят и не разворачивают лишний интерфейс;
6. animations не отвлекают и не создают visual noise;
7. mobile-версия сохраняет быстрый путь к покупке;
8. карточка лучше отвечает на вопросы:
   - сколько стоит;
   - есть ли в наличии;
   - как купить;
   - можно ли доверять товару;
   - какие у него ключевые свойства.

## 13. Technical acceptance criteria

После реализации обязательно:

- `npm run check`
- `npm run build`
- `npm test`

И вручную проверить:

- товар с рейтингом;
- товар без рейтинга;
- товар с oldPrice;
- товар без oldPrice;
- товар с ограниченным остатком;
- товар с несколькими магазинами;
- товар без описания;
- товар с длинными характеристиками;
- mobile viewport;
- desktop viewport.

## 14. Recommended next step

Следующий правильный шаг:

1. реализовать `Phase 1 — layout foundation`;
2. прогнать browser-based visual smoke;
3. затем перейти к `Phase 2` и только потом к motion polish.

Это позволит сначала стабилизировать архитектуру карточки, а уже потом добавлять анимации и декоративные детали.
