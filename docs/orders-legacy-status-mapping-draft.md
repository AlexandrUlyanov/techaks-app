# Orders Legacy Status Mapping Draft

## Назначение

Это только draft-карта соответствия legacy-статусов новой модели заказов.

Важно:

- ничего из этого документа не применяется автоматически;
- документ не является разрешением на массовую миграцию статусов;
- если статус неизвестен, он должен идти в `unknown` / `problematic`, а не угадываться.

## Риски массовой миграции legacy-статусов

Массовая миграция статусов опасна, потому что:

- один и тот же legacy-статус мог использоваться в разном бизнес-контексте;
- часть legacy-значений могла фактически обозначать оплату, а не жизненный цикл заказа;
- старые заказы могут не содержать достаточно контекста для безопасного преобразования;
- неверная миграция статуса способна сломать фильтры, отчёты и работу менеджеров.

## Что делать безопасно

До любой миграции статусов нужно:

1. снять реальное распределение `orders.status`;
2. отделить order-status от payment-status;
3. подтвердить карту соответствия на данных production;
4. прогнать mapping на staging;
5. получить отдельное подтверждение на применение.

## Что не делать

Без отдельного подтверждения нельзя:

- массово переписывать `orders.status`;
- автоматически переводить неизвестные значения в “похожий” новый статус;
- смешивать payment-status и order-status;
- применять draft mapping как rollout-инструкцию.
