# ТЗ: каталог Techaks как навигационная система категорий

## 1. Цель

Перестроить каталог Techaks так, чтобы он работал по двум разным сценариям:

1. Навигационный сценарий — когда пользователь еще не выбрал точный тип товара и ему нужно найти правильный раздел.
2. Покупательский сценарий — когда пользователь уже пришел в конечную категорию и хочет быстро отфильтровать и купить товар.

Главная задача: перестать показывать родительские категории как обычный список товаров, если внутри них есть дочерние разделы.

Каталог должен стать более понятным, быстрее вести пользователя к нужной ветке и при этом сохранить сильный коммерческий сценарий в конечных категориях.

---

## 2. Основание для изменений

Анализ конкурентов показывает устойчивый паттерн:

- маркетплейсы вроде AliExpress используют верхние категории как промо-навигационные лендинги;
- крупные ритейлеры вроде М.Видео и DNS используют конечные категории как классические товарные листинги с фильтрами;
- у retail-каталогов верхний уровень помогает выбрать направление, а не сразу фильтровать товары.

Для Techaks это означает следующее:

- корень каталога и промежуточные категории должны помогать выбрать раздел;
- конечные категории должны помогать купить;
- эти два режима нельзя смешивать в одном UX-паттерне.

---

## 3. Бизнес-цели

1. Сократить количество тупиковых сценариев, когда пользователь попадает в родительскую категорию, но не понимает структуру раздела.
2. Повысить переходы из верхних и промежуточных категорий в конечные категории.
3. Снизить когнитивную нагрузку в крупных разделах каталога.
4. Сохранить или улучшить конверсию в конечных товарных категориях.
5. Сделать каталог пригодным для дальнейшего роста структуры без хаоса в UI.

---

## 4. Ключевые UX-принципы

1. Если у категории есть дочерние категории, она показывается как навигационный экран.
2. Если у категории нет дочерних категорий, она показывается как товарный листинг.
3. Пользователь должен понимать, где он находится в дереве каталога.
4. Главный CTA на навигационном экране — выбрать подкатегорию, а не открыть все товары.
5. Категории не должны дублироваться одновременно в нескольких одинаковых по смыслу блоках.
6. Визуальный стиль карточек категорий и товаров должен быть согласован: без прыжков, без сильных теней, без тяжелых hover-эффектов.
7. Mobile-опыт не должен быть уменьшенной копией desktop. Для мобильного каталога нужен свой вертикальный сценарий.

---

## 5. Область работ

В scope входят:

- `/catalog?cat=all`
- все родительские категории, у которых есть дочерние категории
- все промежуточные подкатегории, у которых есть дочерние категории
- конечные категории как отдельный режим отображения
- desktop + mobile UX каталога
- поиск по категориям
- дерево категорий / accordion
- category cards
- логика переходов и URL
- SEO и accessibility для страниц категорий

Вне scope:

- изменение структуры URL
- изменение бизнес-логики корзины и checkout
- переработка карточки товара на PLP beyond catalog scope
- изменение структуры синхронизации категорий с внешними источниками, кроме необходимых API для preview

---

## 6. Логика отображения по уровням

### 6.1. Корень каталога

Пример: `/catalog?cat=all`

Экран должен показывать:

- хлебные крошки
- заголовок раздела
- поиск по категориям
- левое дерево крупных категорий
- справа карточки верхнего уровня или карточки выбранной ветки

Экран не должен:

- показывать тяжелый баннер
- быть перегружен товарной выдачей
- дублировать категории одновременно в чипсах, карточках и sidebar

### 6.2. Родительская категория

Пример: `/catalog?cat=krasota-i-zdorove`

Если категория содержит дочерние категории, она должна повторять поведение корня каталога, но в рамках своего раздела.

Показываем:

- breadcrumbs
- H1
- короткое описание раздела
- поиск по категориям внутри раздела
- карточки дочерних категорий
- sidebar только если подкатегорий много
- вторичную кнопку `Показать все товары в разделе`

### 6.3. Промежуточная категория

Пример: `/catalog?cat=uhod-za-polostyu-rta`

Если внутри есть дочерние категории:

- не открывать сразу товарный листинг
- не выбирать автоматически первую дочернюю категорию
- активной считать именно текущую категорию
- показывать все дочерние категории как варианты выбора

### 6.4. Конечная категория

Пример: `/catalog?cat=zubnye-shchetki`

Если дочерних категорий нет:

- показываем обычную товарную выдачу
- фильтры
- сортировка
- карточки товаров
- наличие
- способ получения
- пагинация / load more в рамках существующей архитектуры

---

## 7. Правила layout

### 7.1. Desktop

#### Если дочерних категорий <= 6

- не показывать левый sidebar
- показывать крупную сетку карточек категорий
- поисковая строка остается сверху
- CTA `Показать все товары в разделе` остается вторичным

#### Если дочерних категорий > 6

- показывать левое дерево категорий
- дерево должно быть sticky
- справа показывать карточки текущей ветки
- на широких экранах не должно оставаться больших пустот

### 7.2. Mobile

- не использовать двухколоночный layout
- сверху: breadcrumbs, H1, описание, поиск
- ниже: быстрые категории и accordion
- несколько accordion-секций могут быть раскрыты одновременно
- без горизонтального скролла категорий, кроме специальных secondary chips, если они действительно нужны

---

## 8. Поиск по категориям

### 8.1. Назначение

Поиск на навигационных страницах каталога ищет прежде всего категории, а не товары.

### 8.2. Требования

- поиск по названию категории
- поиск по дочерним и родительским категориям
- частичные совпадения
- без учета регистра
- нормализация `е/ё`
- возможность расширения до словаря синонимов

### 8.3. Поведение

- результаты появляются без перезагрузки страницы
- debounce 150–250 ms
- keyboard navigation на desktop:
  - ArrowDown
  - ArrowUp
  - Enter
  - Escape

### 8.4. Результат поиска

Каждый результат должен содержать:

- название категории
- путь внутри каталога
- productCount, если есть
- preview image или иконку
- стрелку перехода

### 8.5. Пустое состояние

Если ничего не найдено:

- сообщение `Ничего не найдено`
- пояснение
- возврат к обычной структуре каталога после очистки поля

---

## 9. Дерево категорий

### 9.1. Desktop sidebar

Требования:

- sticky
- max-height по viewport
- внутренний scroll только если реально нужно
- активная ветка выделена
- вложенность понятна визуально
- длинные названия не должны быть непонятно обрезаны

### 9.2. Mobile accordion

Требования:

- плавное раскрытие
- можно раскрыть несколько разделов сразу
- строки не меньше 44px по высоте
- активность должна быть понятна не только цветом

---

## 10. Карточки категорий

### 10.1. Общие требования

- карточка целиком кликабельна
- единый стиль для корня каталога и промежуточных категорий
- без сильных теней
- без движения картинки при hover
- без сдвига контента
- hover только через мягкую обводку / ring

### 10.2. Leaf category card

Показываем:

- изображение или иконку
- название категории
- количество товаров или fallback-текст
- стрелку перехода

### 10.3. Parent category card

Показываем:

- изображение или иконку
- название
- до 3–4 дочерних названий
- понятный сигнал, что внутри есть структура, а не только товары

### 10.4. Визуальный стиль

- светлый фон / адаптированная темная тема
- скругление по текущей дизайн-системе
- без подъема на hover
- без масштабирования изображения
- без появления тяжелой тени

---

## 11. Secondary content на навигационных категориях

Допускается вторичный контент ниже основного навигационного блока:

- популярные товары раздела
- новинки
- хиты
- подборки

Правила:

- только после блока выбора категорий
- не выше поиска и дерева
- не должен конкурировать с выбором подкатегории
- если данных мало или блок бесполезен — не рендерить

---

## 12. Логика данных и API

### 12.1. Базовая модель

```ts
type CategoryNode = {
  id: string;
  slug: string;
  name: string;
  parentId: string | null;
  description?: string | null;
  imageUrl?: string | null;
  iconUrl?: string | null;
  productCount?: number;
  children?: CategoryNode[];
  isLeaf: boolean;
};
```

### 12.2. Preview endpoint

Для навигационных экранов нельзя тянуть всю товарную выдачу ради превью.

Нужен lightweight источник данных для category preview:

- categoryId
- productCount
- previewImage
- previewImageVariants
- hasChildren

### 12.3. Правило выбора режима

```ts
if (currentCategory.children.length > 0) {
  return <CategoryNavigationPage />;
}

return <ProductListingPage />;
```

---

## 13. SEO-требования

Для каждой категории:

- уникальный `title`
- уникальный `description`
- H1 соответствует категории
- breadcrumbs индексируемые
- ссылки на дочерние категории — обычные `<a>` или equivalent Link
- страница не должна быть JS-only навигацией без crawlable ссылок

Для навигационных категорий:

- текстовый вводный блок должен быть коротким, но достаточным для понимания темы раздела
- не генерировать SEO-спам с переспамом категорий

---

## 14. Accessibility

Обязательно:

- label у поля поиска
- aria-label у кнопки очистки
- aria-expanded у accordion
- видимый focus state
- кликабельные элементы не менее 44px на mobile
- активная категория должна отличаться не только цветом

---

## 15. Dark theme

Все навигационные страницы каталога должны корректно работать в dark theme:

- заголовки и тексты не должны проваливаться по контрасту
- sidebar должен оставаться читаемым
- карточки категорий не должны сливаться с фоном
- hover-ring должен быть виден и в светлой, и в темной теме
- изображения не должны теряться на темном фоне

---

## 16. Пустые и проблемные состояния

### 16.1. Loading

- skeleton для дерева
- skeleton для карточек
- без layout jumps

### 16.2. Empty

Если нет дочерних категорий и нет товаров:

- сообщение
- ссылка назад в каталог

### 16.3. Error

Если не удалось загрузить структуру:

- ошибка
- кнопка повторить

---

## 17. Аналитика

Нужно фиксировать события:

- открытие навигационной категории
- поиск по категориям
- клик по карточке категории
- клик по пункту дерева
- переход в `Показать все товары в разделе`
- переход из navigation category в leaf category

Метрики:

- CTR карточек категорий
- CTR sidebar links
- search usage rate
- search success rate
- переходы в конечные категории
- bounce rate у родительских категорий до и после релиза

---

## 18. Этапы внедрения

### Этап 1. Базовая архитектура

- определить и формализовать режим category navigation vs product listing
- привести корень каталога к стандарту
- унифицировать category cards

### Этап 2. Промежуточные категории

- перевести крупные родительские категории на navigation mode
- внедрить compact mode и sidebar mode
- доделать mobile accordion

### Этап 3. Search + preview data

- стабилизировать поиск по категориям
- добавить lightweight preview источник
- убрать лишнюю зависимость от полной товарной выдачи

### Этап 4. SEO + analytics + polish

- SEO metadata
- breadcrumbs
- accessibility
- analytics
- dark theme polish

---

## 19. Acceptance Criteria

Работа считается выполненной, если:

1. `/catalog?cat=all` работает как навигационная страница, а не как товарная выдача.
2. Любая категория с дочерними категориями открывается как navigation page.
3. Любая конечная категория открывается как product listing.
4. На страницах с малым числом подкатегорий sidebar скрывается.
5. На страницах с большим числом подкатегорий sidebar отображается.
6. Поиск по категориям работает без перезагрузки.
7. Карточки категорий в каталоге имеют единый спокойный hover без масштабирования и теней.
8. Desktop и mobile сценарии работают по разным, но согласованным UX-правилам.
9. Страницы категорий выглядят корректно в dark theme.
10. SEO-метаданные и breadcrumbs не ломаются.
11. Нет больших пустых зон из-за неправильной композиции карточек.
12. Нет дублей одних и тех же категорий в нескольких равнозначных блоках.

---

## 20. Ожидаемый результат

После завершения работ каталог Techaks должен восприниматься как:

- понятная карта магазина на верхних уровнях;
- быстрый коммерческий инструмент на конечных уровнях;
- аккуратная, современная иерархия без перегруза;
- система, которую можно масштабировать без UX-долга.
