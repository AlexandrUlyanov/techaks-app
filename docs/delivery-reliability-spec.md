# ТЗ: надёжный контур расчёта и исполнения доставки
## 1. Цель

Сделать доставку Techaks устойчивой к подмене цены на клиенте, повторным запросам,
таймаутам API перевозчика, перезапуску приложения и несвоевременной оплате заказа.
Менеджер должен видеть, что произошло с вызовом курьера, и уметь безопасно повторить
операцию без риска создать вторую заявку.

## 2. Границы работ

В контур входят:

- расчёт тарифа в checkout;
- фиксация серверной котировки;
- создание заказа с доверенной стоимостью доставки;
- автоматический вызов курьера по статусу `Передан в доставку`;
- синхронизация статуса перевозчика;
- ручная диагностика и повтор из админки;
- миграция, мониторинг, тестирование и откат.

Не входят: маршрутизация нескольких отправлений одного заказа, возвратная логистика,
собственная курьерская служба и динамическая наценка на тариф.

## 3. Термины и источники истины

- **Котировка** — рассчитанное сервером предложение перевозчика, действующее 10 минут.
- **Источник отправления** — склад/магазин, выбранный текущей логикой комплектации.
- **Задача доставки** — идемпотентная фоновая операция вызова курьера.
- Цена товара — только из БД на момент оформления.
- Цена доставки — только из активной серверной котировки.
- Статус заказа — МойСклад после успешной синхронизации.
- Статус перевозки — API Яндекс Доставки.

## 4. Сценарий расчёта

1. Клиент выбирает доставку, подтверждённый GeoSuggest адрес и состав корзины.
2. Backend повторно загружает товары, цены, остатки и склад комплектации.
3. Backend запрашивает тариф у перевозчика.
4. При наличии цены создаётся `delivery_quotes` со сроком жизни 10 минут.
5. Клиент получает `quoteId`, цену, ETA и источник отправления, но не может изменить
   серверную запись.
6. При изменении адреса, состава корзины, количества, цены или склада fingerprint
   меняется, старая котировка становится непригодной.
7. При оформлении backend повторно строит fingerprint. Только совпавшая активная
   котировка участвует в итоговой сумме.
8. Котировка потребляется атомарно в транзакции создания заказа.

Если тариф не найден, заказ разрешено оформить с нулевой стоимостью доставки для
последующего ручного расчёта менеджером. Перед онлайн-оплатой клиент видит, что
стоимость уточняется; клиентская цена никогда не принимается.

## 5. Сценарий вызова курьера

Триггеры:

- статус заказа в админке изменён на `handed_to_delivery`;
- такой статус пришёл webhook МойСклад;
- статус обнаружен массовой пересинхронизацией;
- reconciliation обнаружил пропущенную заявку.

Каждый триггер создаёт или актуализирует задачу с ключом `dispatch:<orderId>`.
Уникальный ключ запрещает дубли. Worker запускается раз в 15 секунд.

Перед обращением к API обязательны проверки:

- тип получения — доставка;
- заказ существует;
- статус соответствует передаче в доставку;
- заказ полностью оплачен: `payment_status=paid` и `paid_amount >= total_price`;
- заявка перевозчика ещё не создана.

Неоплаченная задача получает состояние `blocked` без расходования попыток. После
оплаты reconciliation возвращает её в очередь. Временные ошибки повторяются с
экспоненциальной задержкой, максимум 8 попыток. После исчерпания задача становится
`dead` и требует ручной проверки.

## 6. Состояния

### Котировки

- `active` — может быть использована;
- `consumed` — связана с заказом;
- `expired` — срок истёк.

### Задачи

- `pending` — ждёт worker;
- `processing` — захвачена worker;
- `blocked` — ждёт полной оплаты;
- `failed` — временная ошибка, запланирован retry;
- `completed` — заявка создана или уже существовала;
- `dead` — попытки исчерпаны.

Зависший `processing` lock старше 5 минут автоматически освобождается.

## 7. Админка

В карточке заказа показывать:

- ID и статус заявки перевозчика;
- локальный статус доставки;
- дату последней синхронизации;
- состояние фоновой задачи;
- число попыток;
- последнюю ошибку;
- кнопки обновления/отмены заявки;
- безопасный повтор через очередь.

Ручной повтор сбрасывает попытки, но не может создать дубль при существующем
`deliveryProviderOrderId`.

## 8. Хранение данных

### `delivery_quotes`

Хранит публичный UUID, fingerprint корзины, склад, адрес назначения, цену, валюту,
ETA, provider offer, raw response, срок действия и связь с заказом.

### `delivery_jobs`

Хранит order ID, тип, состояние, idempotency key, попытки, lock, run-after,
последнюю ошибку и диагностический payload.

В `orders.delivery_quote_id` сохраняется котировка, использованная при оформлении.

## 9. Безопасность

- Клиентские `totalPrice` и `deliveryPrice` не являются источником истины.
- UUID котировки проверяется вместе с fingerprint.
- OAuth-токены перевозчика не попадают на frontend, в raw quote и audit.
- Ошибки API показываются менеджеру без секретов.
- Повторные запросы защищены уникальным idempotency key.

## 10. Наблюдаемость

Обязательные сигналы:

- количество `pending/failed/blocked/dead` задач;
- возраст старейшей незавершённой задачи;
- число заказов `handed_to_delivery` без provider order ID;
- доля истёкших котировок;
- ошибки worker в PM2 log.

Критические условия:

- любая `dead` задача;
- `failed` старше 30 минут;
- заказ передан в доставку без заявки более 10 минут;
- `processing` lock старше 5 минут.

## 11. Миграция и совместимость

Миграция только additive:

- добавить nullable `orders.delivery_quote_id`;
- создать `delivery_quotes`;
- создать `delivery_jobs`;
- создать индексы.

Скрипт миграции идемпотентный и выполняется до сборки/restart. Старые заказы работают
без quote ID. Reconciliation подбирает старые переданные в доставку заказы.

## 12. Критерии приёмки

- подмена `deliveryPrice` не меняет сумму заказа;
- изменённая корзина не принимает старую котировку;
- одна котировка не создаёт два заказа;
- неоплаченный заказ не вызывает курьера;
- после оплаты заблокированная задача успешно продолжается;
- timeout API не теряет операцию;
- перезапуск приложения освобождает устаревший lock;
- повторные webhook не создают вторую заявку;
- менеджер видит задачу и ошибку на русском языке;
- production schema verifier подтверждает все таблицы и колонки.
