# ТЗ: Blog 2.0 и AI-помощник редактора для ТЕХАКС

Дата: 2026-05-21  
Статус: draft / planned  
Область: storefront content, admin CMS, SEO, AI assistant, editorial workflow

## 1. Контекст

В проекте уже есть рабочий базовый блог:

- публичный список статей;
- публичная страница статьи;
- админский CRUD для статей;
- простая таблица `posts`;
- вывод последних статей на главной.

Это уже достаточно для публикации материалов, но пока недостаточно для сильного контентного раздела интернет-магазина.

Текущие ограничения:

- контент статьи рендерится как сырой HTML/text без зрелого editorial pipeline;
- нет расширенного SEO-слоя для статьи;
- нет нормальной редакционной модели со статусами, авторами и временем публикации;
- нет связанных статей, тегов, featured-логики и контентной аналитики;
- админский редактор пока утилитарный, а не редакционный;
- нет AI-помощника, который помогает писать, улучшать и SEO-оптимизировать статьи.

Нужно спроектировать и внедрить следующий уровень блога:

1. безопасный и удобный редакторский контур;
2. более сильную витрину блога;
3. SEO-слой для статей;
4. AI-помощника в админке;
5. связку блога с ассортиментом и коммерческими сценариями магазина.

## 2. Цель

Сделать так, чтобы блог:

1. стал полезным контентным разделом для клиентов;
2. усиливал SEO и доверие к магазину;
3. помогал продавать через полезные обзоры, гайды и подборки;
4. управлялся через удобную редакторскую админку;
5. использовал AI как помощника редактора, а не как бесконтрольный генератор текста.

## 3. Главные принципы

### 3.1 Блог — это storefront-инструмент, а не просто CMS-таблица

Статьи должны работать на:

- доверие;
- organic traffic;
- коммерческие сценарии выбора товара;
- связку с категориями, брендами и ассортиментом.

### 3.2 AI помогает редактору, но не публикует сам

AI-слой должен:

- предлагать;
- улучшать;
- структурировать;
- объяснять;

но не должен бесконтрольно:

- переписывать готовый материал без подтверждения;
- автоматически публиковать статьи;
- генерировать “SEO-мусор” ради количества.

### 3.3 Сначала безопасность и структура, потом automation

До сильного AI-слоя нужно сначала обеспечить:

- безопасный рендер контента;
- удобный editor flow;
- расширенную модель данных поста;
- прозрачные статусы статьи.

### 3.4 Блог должен говорить языком ТЕХАКС

Контент должен быть:

- понятным;
- практичным;
- связанным с реальными товарами;
- ориентированным на выбор аксессуаров и техники;
- полезным, а не абстрактно “контентным”.

## 5. Что считаем проблемой

### 5.1 Контентный слой пока небезопасный и хрупкий

Сейчас статья рендерится через `dangerouslySetInnerHTML`, а контент по сути хранится как сырой HTML/text.

Это создаёт риски:

- XSS;
- неконсистентная разметка;
- слабый editor UX;
- сложность дальнейшего расширения.

### 5.2 Слишком бедная модель поста

Посту не хватает полей:

- `status`
- `publishedAt`
- `updatedAt`
- `authorName`
- `metaTitle`
- `metaDescription`
- `ogImage`
- `featured`
- `readingTime`
- `tags`

### 5.3 Нет редакционного workflow

Сейчас публикационная модель слишком плоская:

- либо опубликовано;
- либо нет.

Нужны более зрелые статусы:

- `draft`
- `scheduled`
- `published`
- `archived`

### 5.4 Витрина статьи слабее, чем могла бы быть

Нет:

- related posts;
- рубрик/фильтрации;
- featured article;
- author block;
- времени чтения;
- last updated;
- article schema;
- stronger article SEO layer.

### 5.5 Нет AI-помощника

Редактор сейчас пишет всё руками:

- заголовок;
- slug;
- excerpt;
- структуру;
- SEO;
- tone of voice;
- идеи для статей.

Это замедляет производство контента и делает качество неровным.

### 6.1 Blog storefront

### 6.2 Editorial admin

- список постов;
- richer editor;
- preview;

- featured controls;

### 6.3 AI editorial assistant

- помогает придумать тему;
- помогает собрать структуру статьи;
- помогает улучшить черновик;
- помогает с SEO;

## 7. Целевая модель данных

## 7.1 Таблица `posts` должна быть расширена

Рекомендуемые поля:

- `id`
- `slug`
- `title`
- `excerpt`
- `content`
- `content_format`
- `category`
- `image`
- `og_image`
- `meta_title`
- `meta_description`
- `author_name`
- `status`
- `featured`
- `reading_time_minutes`
- `published_at`
- `updated_at`
- `created_at`

### Пояснения

- `content_format`:
  - `html`
  - `structured_json`
  - на старте можно оставить `html`, но предусмотреть расширение
- `status`:
  - `draft`
  - `scheduled`
  - `published`
  - `archived`
- `featured`:
  - для главной, рубрики или editorial акцента

## 7.2 Категории и теги

Во второй фазе рекомендуется завести:

- `post_tags`
- `post_tag_assignments`

Категорию можно на первой фазе оставить строкой, но лучше готовить систему к более нормализованной таксономии.

## 7.3 AI-сущности

Для полноценного AI-слоя рекомендуется добавить:

- `blog_ai_runs`
- `blog_ai_suggestions`

Минимальный состав:

### `blog_ai_runs`

- `id`
- `post_id` nullable
- `mode`
- `prompt_version`
- `status`
- `input_snapshot`
- `result_json`
- `created_by_user_id`
- `created_at`

### `blog_ai_suggestions`

- `id`
- `run_id`
- `post_id` nullable
- `suggestion_type`
- `content`
- `metadata_json`
- `status`
- `applied_at`
- `created_at`

Где `suggestion_type` может быть:

- `title`
- `slug`
- `excerpt`
- `outline`
- `rewrite`
- `seo_meta`
- `cta`
- `ideas`
- `internal_links`

## 8. Безопасность контента

Это первая обязательная часть.

### 8.1 Что нужно сделать

1. Ввести sanitization HTML перед сохранением и/или перед выводом.
2. Явно ограничить разрешённые теги и атрибуты.
3. Убрать возможность произвольного встроенного script/event-handler content.
4. Подготовить путь к structured editor format в будущем.

### 8.2 Что нельзя делать

- нельзя оставлять сырой HTML без sanitization;
- нельзя доверять контенту статьи просто потому, что он введён в админке;
- нельзя тащить arbitrary embed без whitelist policy.

## 9. Public storefront UX

## 9.1 Список статей

Страница `/blog` должна поддерживать:

- featured article сверху;
- сетку статей;
- фильтр по рубрике;
- состояние “статей пока нет”;
- понятную дату;
- excerpt;
- понятный CTA.

### Вторая очередь

- сортировка;
- “популярное / новое”;
- поиск по статьям;
- infinite/paginated archive.

## 9.2 Карточка статьи

Страница статьи должна включать:

- breadcrumbs;
- рубрику;
- title;
- meta block:
  - дата публикации
  - дата обновления
  - автор
  - время чтения
- hero image;
- контент;
- related posts;
- CTA к релевантным товарам/категориям;
- lead/help block без перегруза.

## 9.3 Related content

Нужно поддержать:

- связанные статьи той же категории;
- статьи того же типа;
- статьи по связанным брендам/категориям товаров.

## 9.4 SEO для статьи

На статье должны быть:

- `metaTitle`
- `metaDescription`
- canonical
- OG title/description/image
- article schema / structured data

## 11. AI-помощник в админке

## 11.1 Главный принцип

AI должен быть встроен прямо в `AdminBlog` как editor assistant.

Это не отдельная “магическая страница ради нейросети”, а реальный рабочий помощник редактора.

## 11.2 UI-модель

В редакторе статьи нужен боковой assistant panel или tab-set:

- `Помощник`
- `SEO`
- `Идеи`
- `Связки с каталогом`

## 11.3 Что должен уметь AI v1

### Создание черновика

- предложить 3–5 заголовков;
- предложить slug;
- предложить excerpt;
- предложить outline статьи;
- предложить CTA.

### Улучшение черновика

- сделать текст понятнее;
- сократить воду;
- разбить на блоки/H2/H3;
- сделать tone ближе к ТЕХАКС;
- улучшить вступление и финал.

### SEO-помощь

- предложить meta title;
- предложить meta description;
- подсветить слабые места структуры;
- предложить внутренние ссылки на категории/товары.

### Генератор тем

- темы по категории товара;
- темы по бренду;
- сезонные темы;
- темы по ценовому сегменту;
- темы “как выбрать / чем отличается / что подойдёт”.

## 11.4 Что должен уметь AI v2

- генерировать статью по brief;
- строить подборки на основе ассортимента;
- предлагать related products;
- предлагать SKU/category links;
- превращать короткий brief в draft article.

## 11.5 Что AI не должен делать

- не публиковать автоматически;
- не затирать текст без подтверждения;
- не менять slug/meta/title молча;
- не вставлять небезопасный HTML;
- не превращать блог в SEO-спамогенератор.

## 11.6 Approval flow

Любое AI-предложение должно:

1. быть явно показано;
2. иметь кнопку:
   - `Применить`
   - `Вставить в черновик`
   - `Отклонить`
3. по возможности показывать diff или preview результата.

## 12. Связка блога с каталогом

Это особенно важно для ТЕХАКС.

Нужно поддержать:

- ссылки на категории;
- ссылки на бренды;
- подборки товаров по теме статьи;
- “товары по теме” в статье;
- AI-предложения тем на основе реального ассортимента.

Примеры:

- “Как выбрать кабель USB-C для быстрой зарядки”
- “Какие аксессуары Hoco удобны для автомобиля”
- “Что подарить из техники и аксессуаров до 1000 ₽”

## 13. Редакционные сценарии

Система должна покрывать:

1. редактор создаёт статью с нуля;
2. редактор использует AI для outline;
3. редактор использует AI для title/excerpt/SEO;
4. редактор планирует публикацию;
5. статья попадает на главную как featured;
6. редактор связывает статью с каталогом;
7. редактор обновляет старую статью и фиксирует `updatedAt`.

## 15. Acceptance criteria

Задача считается реализованной качественно, если:

1. статьи больше не рендерятся небезопасно без sanitization;
2. редактор поста имеет richer model, а не только primitive CRUD;
3. у статьи есть SEO-поля и editorial status;
4. блог на витрине стал сильнее как контентный раздел;
5. AI-помощник встроен в админку и помогает редактору без автопубликации;
6. AI умеет предлагать title/slug/excerpt/outline/SEO;
7. редактор может принять или отклонить AI-предложение;
8. блог связан с ассортиментом и коммерческими сценариями магазина;
9. проект проходит:
   - `npm run check`
   - `npm run build`
   - `npm test`

## 16. Что не нужно делать грубо

Не надо:

- оставлять `dangerouslySetInnerHTML` без защиты;
- строить AI как “кнопку написать всё за меня и сразу опубликовать”;
- добавлять сложный heavyweight editor без UX-цели;
- генерировать бессмысленный SEO-контент ради количества;
- разрывать блог и каталог как две отдельные вселенные.

## 17. Рекомендуемый первый production scope

Если идти разумно, первый сильный production-пакет должен включать:

- content sanitization;
- richer post model;

- editorial statuses;
- meta/OG fields;
- improved article page;
- AI assistant v1 в админке:
  - title

Это уже даст взрослый и полезный результат без лишнего раздувания системы.
