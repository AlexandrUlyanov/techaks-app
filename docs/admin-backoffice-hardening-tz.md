# ТЗ: доведение админки Techaks до цельного боевого backoffice

Дата: 2026-06-02

## 1. Контекст

Админка Techaks уже покрывает основные операционные сценарии:

- заказы и карточку заказа;
- резервы;
- товары;
- магазины;
- акции;
- блог;
- поиск;
- отзывы;
- синхронизацию с МойСклад;
- настройки сайта, интеграций и оплаты.

При этом система развивается неравномерно:

- часть экранов уже приведена к новому более зрелому административному UX;
- часть экранов всё ещё живёт в старом визуальном и структурном подходе;
- отдельные разделы заявлены как рабочие, но фактически недоделаны до конца;
- нет единого журнала изменений;
- dashboard не опирается на реальные бизнес-метрики;
- интеграционный контур МойСклад уже рабочий, но под нагрузкой показывает признаки нестабильности.

Нужно довести админку до состояния цельного, предсказуемого и безопасного рабочего инструмента.

## 2. Цель

Сделать админку Techaks:

- функционально завершённой в ключевых административных разделах;
- визуально согласованной;
- более прозрачной для команды;
- более безопасной в части изменений;
- более устойчивой в части синхронизации с МойСклад;
- пригодной для ежедневной работы операторов, менеджеров, контент-команды и администраторов.

## 3. Область работ

В эту задачу входят шесть направлений:

1. Доделать `Категории` до полноценного CRUD.
2. Переделать `Dashboard` на реальные метрики без хардкода.
3. Привести `Stores / Banners / Categories / Dashboard` к единому новому стилю админки.
4. Убрать остаточные Telegram / WhatsApp поля из настроек сайта, если окончательно остаёмся только на внутренней переписке через сайт.
5. Добавить единый `audit log / журнал изменений`.
6. Усилить устойчивость `MoySklad sync` под нагрузкой.

## 4. Направление 1. Категории: полноценный CRUD

### 4.1. Проблема

Раздел категорий сейчас в основном работает как просмотр дерева. Кнопки создания/редактирования/удаления визуально присутствуют, но полноценный рабочий CRUD до конца не реализован.

Это создаёт разрыв между ожиданием и реальной функциональностью.

### 4.2. Цель

Сделать `Админка → Каталог → Категории` полноценным инструментом управления структурой каталога.

### 4.3. Что должно поддерживаться

- создание категории;
- редактирование категории;
- смена родителя;
- изменение сортировки;
- активация / деактивация;
- безопасное удаление;
- просмотр дерева категорий;
- работа с slug;
- опционально SEO-поля и изображение категории.

### 4.4. Минимальный набор полей категории

Использовать текущую модель категории и при необходимости расширить её минимально:

- `id`
- `name`
- `slug`
- `parentId`
- `sortOrder`
- `isActive`
- `image` / `imageUrl` (если уже поддерживается витриной)
- `seoTitle` (опционально)
- `seoDescription` (опционально)
- `createdAt`
- `updatedAt`

### 4.5. UI-раздел

Экран должен включать:

- верхний `AdminPageHeader`;
- дерево категорий;
- primary action `Создать категорию`;
- inline-actions или contextual menu:
  - `Редактировать`
  - `Переместить`
  - `Скрыть / Показать`
  - `Удалить`
- индикаторы:
  - активна ли категория;
  - есть ли дочерние категории;
  - есть ли товары;
  - slug;
  - опционально количество товаров.

### 4.6. Форма создания / редактирования категории

Поля:

- Название
- Slug
- Родительская категория
- Порядок сортировки
- Активна
- Изображение
- SEO title
- SEO description

Поведение:

- slug по умолчанию генерируется из названия;
- если пользователь меняет slug вручную, автогенерация перестаёт переписывать значение;
- slug должен быть уникальным;
- при конфликте показывать понятную ошибку;
- если родитель выбран, нельзя допускать циклические связи.

### 4.7. Удаление категории

Правила:

- если у категории есть дочерние категории — прямое удаление запрещено;
- если к категории привязаны товары — прямое удаление запрещено;
- в этом случае показывать безопасное сообщение:
  - сначала перенесите товары;
  - сначала переместите или удалите дочерние категории;
- для большинства сценариев предпочтительнее soft-disable через `isActive = false`.

Жёсткое удаление можно оставить только для `super_admin`, если бизнес-логика это допускает.

### 4.8. Backend / API

Доделать или добавить методы:

- `category.listTree`
- `category.create`
- `category.update`
- `category.delete`
- `category.reorder`
- `category.move`

### 4.9. Права доступа

- просмотр: `admin`, `super_admin`, `content_manager`, `merchandiser`
- изменение: `admin`, `super_admin`
- удаление: `admin`, `super_admin`, при необходимости только `super_admin`

### 4.10. Acceptance criteria

Готово, если:

- категорию можно создать, отредактировать, переместить и скрыть;
- дерево категорий корректно отображается;
- slug валидируется и уникален;
- нельзя сломать структуру дерева;
- нельзя удалить категорию с товарами или детьми без понятного сценария;
- все действия пишутся в audit log.

## 5. Направление 2. Dashboard: реальные метрики вместо хардкода

### 5.1. Проблема

Текущий dashboard частично использует фиктивные или демонстрационные значения и не отражает реального состояния бизнеса и операционного контура.

### 5.2. Цель

Превратить dashboard в реальный входной экран админки, который помогает быстро понять:

- где сейчас проблемы;
- какие процессы требуют реакции;
- как чувствуют себя заказы, каталог и синхронизации.

### 5.3. Что убрать

Убрать весь хардкод:

- фиктивные тренды;
- фиктивных посетителей;
- фиктивные числа “за месяц”;
- фиктивную “заполняемость БД”;
- любые псевдометрики без реального источника данных.

Если для метрики нет данных — не подставлять красивую цифру. Лучше:

- показать `Нет данных`;
- скрыть карточку;
- или явно подписать, что источник пока не подключён.

### 5.4. Реальные метрики

#### Заказы

- заказов сегодня;
- заказов за 7 дней;
- заказов со статусом `новый / в обработке / проблемный`;
- оплаченных заказов сегодня;
- сумма оплаченных заказов за 7 дней.

#### Каталог

- всего товаров;
- активных товаров;
- скрытых вручную;
- автоматически скрытых из-за отсутствия цены / системной блокировки.

#### Резервы и магазины

- активных резервов;
- резервов, которые истекают сегодня;
- количество активных магазинов.

#### Синхронизации

- статус последнего full sync;
- количество ошибок в queue;
- количество pending jobs;
- наличие stale lock / watchdog problem;
- состояние order sync.

#### Коммуникации

- заказов с непрочитанными сообщениями клиента;
- отзывов на модерации.

### 5.5. Архитектура данных

Сделать отдельный агрегирующий endpoint:

- `admin.dashboard.getOverview`

Он должен возвращать собранную структуру overview, чтобы frontend не тянул десятки несвязанных запросов.

### 5.6. UI dashboard

Структура:

1. Summary band
2. Заказы и деньги
3. Каталог
4. Синхронизации
5. Коммуникации и alerts

Важно:

- не делать ornamental dashboard;
- не добавлять графики только ради графиков;
- на первом экране должны быть именно actionable numbers.

### 5.7. Acceptance criteria

Готово, если:

- dashboard больше не содержит фиктивных чисел;
- каждая карточка опирается на реальные данные;
- отсутствующие данные отображаются честно;
- дашборд быстро открывается и читается за 5–10 секунд.

## 6. Направление 3. Унификация UI: Stores / Banners / Categories / Dashboard

### 6.1. Проблема

Эти разделы всё ещё выглядят как экраны старого поколения админки и выбиваются из нового визуального языка.

### 6.2. Цель

Привести их к единой новой системе административного интерфейса.

### 6.3. Общие принципы нового стиля

- меньше тяжёлых карточек;
- меньше старых теней;
- меньше визуального шума;
- единые `AdminPageHeader`, `AdminSection`, `AdminStatCard`;
- единые вертикальные ритмы;
- единые паттерны для:
  - фильтров,
  - списков,
  - action bar,
  - модалок,
  - empty state,
  - loading state,
  - danger zone.

### 6.4. Обязательные изменения

Для каждого из разделов:

#### Dashboard

- переделать старые бордерные карточки;
- привести layout к новой системе summary + sections;
- встроить реальные метрики.

#### Categories

- убрать старый “серый tree UI”;
- привести к единой иерархии страницы;
- встроить CRUD-формы и одинаковые кнопки действий.

#### Stores

- привести список магазинов к новому UI;
- унифицировать карточку магазина;
- привести форму магазина к единому паттерну;
- аккуратно вписать привязку к складу МойСклад.

#### Banners

- привести список акций к новому стилю;
- привести форму баннера к тому же паттерну, что и другие admin forms;
- сделать состояние черновика, активной записи и удаления единообразным.

### 6.5. Что нужно убрать

- тяжёлые старые тени;
- случайные старые hover-эффекты;
- несистемные радиусы;
- разный визуальный язык у primary actions;
- старые “панели в панели”.

### 6.6. Acceptance criteria

Готово, если:

- четыре раздела выглядят как части одной системы;
- визуальная иерархия едина;
- одинаково работают формы, списки и action blocks;
- нет ощущения, что часть страниц “старая”, а часть “новая”.

## 7. Направление 4. Убрать остаточные Telegram / WhatsApp поля

### 7.1. Условие

Если окончательный курс — только внутренняя переписка на сайте плюс телефон, то Telegram / WhatsApp должны исчезнуть не только с витрины, но и из admin settings.

### 7.2. Что убрать

Из `site profile settings`:

- `whatsappUrl`
- `telegramUrl`
- `telegramHandle`

### 7.3. Что проверить

Перед удалением убедиться, что эти поля:

- не участвуют в публичном профиле сайта;
- не используются в письмах;
- не используются в личном кабинете;
- не выводятся в футере / контактах / legal pages.

### 7.4. Как делать безопасно

Предпочтительно:

- убрать поля из UI;
- убрать поля из публичного ответа настроек;
- в БД оставить поля как deprecated, если это дешевле и безопаснее;
- не ломать старые записи и миграции без необходимости.

### 7.5. Acceptance criteria

Готово, если:

- в админке больше нет Telegram/WhatsApp полей;
- на сайте они больше не выводятся;
- письма и ЛК не ссылаются на внешние мессенджеры;
- основной канал коммуникации — внутренняя переписка и телефон.

## 8. Направление 5. Единый audit log / журнал изменений

### 8.1. Проблема

Сейчас логи есть точечно: в заказах, sync, части настроек. Но нет единого административного журнала, который показывает, кто и что менял в системе.

### 8.2. Цель

Добавить единый, фильтруемый, безопасный журнал административных действий.

### 8.3. Что логировать

#### Каталог

- создание/изменение/удаление категории;
- изменение товара;
- изменение магазина;
- изменение баннера.

#### Заказы

- смена статуса;
- изменение состава;
- изменение оплаты;
- изменение доставки;
- комментарии менеджера;
- сообщения клиенту.

#### Пользователи и доступ

- изменение роли;
- изменение статуса;
- impersonation;
- удаление пользователя.

#### Настройки

- изменения интеграций;
- изменения YooKassa;
- изменения auth settings;
- изменения site profile;
- maintenance mode.

#### Синхронизации

- запуск full sync;
- retry jobs;
- wipe catalog;
- изменение scheduler/runtime settings;
- reconcile;
- ручные операционные действия.

### 8.4. Структура записи

Новая таблица, например:

- `admin_audit_logs.id`
- `actorUserId`
- `actorEmail`
- `actorRole`
- `action`
- `entityType`
- `entityId`
- `entityLabel`
- `beforeJson`
- `afterJson`
- `metaJson`
- `ip`
- `createdAt`

### 8.5. Безопасность

Нельзя писать в журнал:

- secret keys;
- SMTP password;
- API tokens целиком;
- encrypted values;
- персональные секреты.

Для чувствительных данных:

- логировать только факт изменения;
- хранить mask / last4 / configured flag.

### 8.6. UI журнала

Новая страница:

- `Админка → Система → Журнал изменений`
  или
- вкладка в `Настройки`

Фильтры:

- период;
- пользователь;
- роль;
- сущность;
- действие.

Список должен показывать:

- кто;
- что;
- когда;
- над чем;
- краткое описание.

Карточка детали:

- before / after;
- metadata;
- ссылка на связанную сущность.

### 8.7. Acceptance criteria

Готово, если:

- основные admin actions попадают в журнал;
- секреты в журнал не записываются;
- журнал фильтруется;
- по записи можно понять, кто изменил объект и что именно изменилось.

## 9. Направление 6. Усиление устойчивости MoySklad sync

### 9.1. Проблема

На проде уже наблюдались:

- `429`
- `503`
- `read ETIMEDOUT`
- `timeout`

Основная причина — наложение full sync, webhook processing и order sync на ограниченное API МойСклад.

### 9.2. Цель

Снизить число transient failures и сделать интеграцию устойчивее под реальной нагрузкой.

### 9.3. Что нужно сделать

#### 9.3.1 Ограничение параллелизма

Ввести более явное управление concurrency:

- отдельно для full sync;
- отдельно для order sync;
- отдельно для webhook queue.

#### 9.3.2 Rate-limit aware retry

Для:

- `429`
- `503`
- `ETIMEDOUT`
- `ECONNRESET`
- `timeout`

нужно:

- exponential backoff;
- jitter;
- лимит числа ретраев;
- понятная классификация ошибок.

#### 9.3.3 Разведение order sync и full sync

Если идёт full sync:

- order sync не должен бомбить API в прежнем темпе;
- допустимы режимы:
  - serial,
  - reduced concurrency,
  - controlled delay.

#### 9.3.4 Кэш metadata

Снизить число повторных запросов к:

- organization;
- store;
- saleschannel;
- states;
- другим стабильным reference-данным.

#### 9.3.5 Улучшить диагностику

В sync logs и queue logs добавить больше контекста:

- endpoint;
- attempt;
- entity type;
- retry delay;
- тип ошибки;
- job id / correlation id.

#### 9.3.6 Разделить timeout strategy

Разные классы операций должны иметь разные таймауты:

- тяжёлые batch/full sync;
- краткие order/payment sync;
- справочные metadata requests.

#### 9.3.7 Проверить идемпотентность

Повторные попытки не должны:

- создавать дубли customerorder;
- создавать дубли paymentin;
- плодить лишние side effects.

### 9.4. UI/операторский контур

На экране синхронизации полезно усилить:

- индикатор давления rate limit;
- состояние “идёт full sync, order sync замедлен”;
- визуальное отделение transient errors от logic/data errors;
- советы оператору по следующему действию.

### 9.5. Acceptance criteria

Готово, если:

- уменьшается число sync ошибок под нагрузкой;
- order sync и full sync меньше мешают друг другу;
- ретраи безопасны;
- журналы понятнее объясняют, что происходит;
- оператор видит не просто ошибку, а operational context.

- TypeScript без ошибок;
