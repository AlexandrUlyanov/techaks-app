# TechAks Workspace Guidelines

This file outlines strict architectural and developmental conventions for AI agents operating within the `techaks-app` workspace.

## 🏗️ Technology Stack
- **Frontend:** React 19, Vite, TailwindCSS, Radix UI primitives.
- **Backend:** Hono (Node.js Adapter), tRPC.
- **Database:** MySQL, Drizzle ORM.
- **Deployment:** Ubuntu VPS (REG.RU) managed by PM2 and Nginx.
- **Production server size:** 1 vCPU, 1 GB RAM, 10 GB disk. Keep maintenance
  jobs sequential and bounded; avoid unbounded catalog-wide work in memory.

## 🛡️ Core Directives

1. **Database Schema & Migrations:**
   - Always use Drizzle ORM (`drizzle-kit`).
   - Run `npm run db:push` to apply schema changes. In CI/CD, the `--force` flag is used.
   - Note: The `slug` column for `products`, `categories`, `banners`, and `posts` has a maximum length of `255` characters.
   - External IDs (like MoySklad UUIDs) are stored in the `msId` (or `ms_id` in DB) column.

2. **MoySklad Synchronization (`api/routers/sync.ts`):**
   - **IDs:** MoySklad often appends query parameters to `href` UUIDs (e.g., `?expand=supplier`). **You MUST strip these parameters** (`.split("?")[0]`) before saving to the DB or attempting to map local entities.
   - **Slugs:** URLs are automatically generated by transliterating the Cyrillic name using the `slugify` utility in `lib/utils.ts`. They are truncated to 200 chars to leave room for uniqueness suffixes.
   - **Images:** Downloaded images MUST be saved to `./public/images/{category-slug}/`. This ensures they are served correctly by Hono's `serveStatic` middleware and survive PM2 process restarts and builds.
   - **Matching:** Local items are matched to MoySklad items via the `msId` column, NOT the slug. This allows the user to manually change a product's slug in the Admin UI without breaking the sync.

3. **Backend Static Serving (`api/boot.ts`):**
   - In production, Vite builds the frontend into `dist/public`, and the backend into `dist/boot.js`.
   - Hono is configured to serve standard frontend assets from `dist/public`.
   - However, dynamically uploaded/synced files (like `/images/*`) are strictly served from `./public`. Do NOT save dynamically generated files inside `dist/`.

4. **CI/CD & Deployment:**
   - Deployment targets an Ubuntu VPS (`195.208.2.100`), not Google Cloud Run.
   - Workflow: `.github/workflows/deploy.yml`. It uses SSH to pull code, run `npm run db:push`, `npm run build`, and `pm2 restart techaks`.
   - Nginx handles port 80 and reverse-proxies to `127.0.0.1:3000`.

5. **Type Safety:**
   - Ensure all tRPC inputs and outputs are strictly typed with `zod`.
   - Avoid `any` unless integrating with poorly typed third-party API responses (like raw MoySklad dumps), but still strive to interface it cleanly.

6. **UI/UX Consistency:**
   - Use Lucide-react for all icons.
   - Breadcrumbs should be standardized in a top bar with a `bg-muted/30` background.
   - Do not use Tailwind `@apply` excessively; prefer utility classes directly in JSX.

7. **Authorization & Security (CASL + JWT):**
   - **Authentication:** Use JWT tokens for user session management. Tokens are passed in the `Authorization: Bearer <token>` header or `token` cookie.
   - **CASL Abilities:** Permissions are centralized in `contracts/ability.ts`. Use `defineAbilityFor(user)` to build an `AppAbility` instance.
   - **tRPC Procedures:** 
     - Use `publicQuery` only for truly public, read-only data.
     - Use `protectedProcedure` for all authenticated requests.
     - Use `adminProcedure` as a shortcut for `manage all` permissions.
     - Within procedures, use `requireAbility(ctx, action, subject)` to perform granular permission checks.
   - **Frontend:** Use `useAbility()` hook or `<Can>` component from `@casl/react` (via `AbilityProvider`) to conditionally show UI elements. Protect routes using the `ProtectedRoute` component in `App.tsx`.
